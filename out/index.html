Why it exists

Windows always had the power. It just never had one coherent interface for agents.

Most automation stacks on Windows force you to choose between brittle UI scripting, app-specific object models, or hand-built shell orchestration. WinScript lets one runtime decide which layer to use, then reports back in agent-friendly language.

One protocol across multiple surfaces

Agents stop switching mental models every time a task moves from visible UI to Office objects to system setup work.

Clear failure modes instead of silent script drift

Structured outcomes make it easier to retry on purpose, recover state, and continue long-running jobs.

Built for operators as much as prompt engineers

The same runtime can support human-run workflows, internal copilots, and fully autonomous desktop loops.

System story

A runtime that speaks to agents on one side and Windows on the other.

The hard part is not clicking buttons. The hard part is knowing when to switch surfaces, how to report failure, and how to keep a multi-step workflow understandable to the next model call.

Runtime Layers

Agent protocol

Tool schemas, typed parameters, predictable responses, and a compact interface that models can reason over quickly.

Decision engine

Routing logic that chooses between UIA, COM, and shell execution while preserving traceability for retries.

Windows reach

The actual desktop substrate: application object models, accessible control trees, process state, and machine-level commands.

01 / interpret

Take agent intent and narrow it into a concrete action plan.

WinScript turns high-level instructions into targetable operations that can survive UI drift, loading states, and missing focus.

02 / route

Pick the right Windows surface instead of forcing one integration path.

The runtime can flow toward UI Automation, COM/OLE, or PowerShell depending on the task, so agents stop guessing which subsystem to trust.

03 / recover

Return structured outcomes that help the next model step recover fast.

Clear errors, screenshots, and state hints make retries useful instead of random. That is what turns automation into an agent loop instead of a brittle macro.

Capability rail

One runtime, four high-value surfaces.

What matters is the shape of the reach: visible UI, deep object models, system actions, and the recovery loop that keeps automation stable when the desktop changes underneath you.

UI Automation

Read windows, inspect controls, click targets, and fill forms across Win32, WPF, WinUI, and Electron surfaces.

Best for desktop navigation and state inspection.

COM and OLE

Reach into Office, legacy line-of-business tools, and apps that still expose deep object models behind the visible UI.

Best for spreadsheet, document, and ERP-style workflows.

Shell and system control

Run PowerShell commands, touch the registry, and orchestrate background tasks when the cleanest path is below the interface.

Best for setup, remediation, and fleet actions.

Recovery surfaces

Give agents the signal they need to retry intentionally with typed failures, screenshots, and action traces.

Best for keeping long automation runs stable.
